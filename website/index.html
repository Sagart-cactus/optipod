Kubernetes resource optimization is broken

Most clusters suffer from guesswork, automation fear, and GitOps conflicts. The result is indefinite over-provisioning or disabled optimization entirely.

Most clusters suffer from

  • CPU and memory requests guessed once and never revisited.
  • VPA conflicting with GitOps workflows.
  • Fear of memory tuning due to OOMKills and restarts.
  • Lack of explainability and auditability in tools.

So teams respond by

  • Over-provisioning indefinitely to stay safe.
  • Disabling automation entirely to avoid surprises.
  • Accepting waste as the cost of stability.
  • Choosing manual tuning that never scales.

Designed for trust, not surprises

OptiPod takes a fundamentally different approach to resource optimization, focused on safety, explainability, and GitOps-first workflows.

GitOps-safe by design

  • Uses Kubernetes Server-Side Apply with field-level ownership.
  • Manages only CPU and memory requests/limits — nothing else.

Explainable decisions

  • Every recommendation includes usage data and margins.
  • Rationale is visible before any action is taken.

Safety-first automation

  • Conservative defaults and configurable bounds.
  • Controlled change rates prevent disruptive updates.
  • Workload type filtering for gradual adoption.

Adopt gradually

  • Start in observe or recommend mode.
  • Enable automatic application only when you’re ready.

What OptiPod Will Not Do

  • OptiPod will never mutate workloads without explicit configuration.
  • OptiPod will never override GitOps ownership.
  • OptiPod will never blindly reduce memory.
  • All changes are observable, auditable, and reversible.

A Kubernetes-native control loop

OptiPod runs entirely inside your cluster, evaluates real usage, and makes reversible, auditable changes when you allow it.

How it works

  1. Discovers workloads matching your policy selectors and workload type filters.
  2. Collects real-time and historical resource usage.
  3. Evaluates usage using strategies like P90, P95, P99 percentiles.
  4. Generates recommendations with safety constraints applied.
  5. Optionally applies changes using Server-Side Apply.
  6. Every action is observable, auditable, and reversible.

No external SaaS dependency. All enforcement happens inside your cluster.

Start safely in recommend mode

No mutations are applied until you explicitly allow them.

Create a policy

apiVersion: optipod.optipod.io/v1alpha1
kind: OptimizationPolicy
spec:
  mode: Recommend
  selector:
    workloadSelector:
      matchLabels:
        optimize: "true"
  metricsConfig:
    provider: metrics-server
    rollingWindow: 24h
    percentile: P90
    safetyFactor: 1.2
  resourceBounds:
    cpu:
      min: "100m"
      max: "4000m"
    memory:
      min: "128Mi"
      max: "8Gi"
  updateStrategy:
    allowInPlaceResize: true
    allowRecreate: false
    updateRequestsOnly: true

Install OptiPod

kubectl apply -f https://github.com/Sagart-cactus/optipod/releases/latest/download/install.yaml
